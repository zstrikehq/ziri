# ZS AI Gateway - Current Project State

**Last Updated**: January 2025  
**Purpose**: This document provides a comprehensive overview of the ZS AI Gateway project's current state, architecture, and implementation details. It is designed to help LLM models understand the project structure and current functionality.

---

## Project Overview

**ZS AI Gateway** is a local proxy server that acts as an authorization gateway for Large Language Model (LLM) API calls. It provides:

1. **Local Proxy Server**: Runs on the admin's machine, manages users, API keys, and LLM provider credentials
2. **Authorization Gateway**: Intercepts LLM requests, validates them through a Policy Decision Point (PDP), and routes authorized requests to LLM providers
3. **Management UI**: Web-based interface for admins to manage users, API keys, providers, and policies
4. **User SDK**: Client library for end-users to make authorized LLM calls through the gateway
5. **Management SDK**: Client library for programmatic management operations

## Authentication & Authorization

### Admin Authentication

**Flow:**
1. Admin opens UI (served by proxy server)
2. Redirected to `/login` if not authenticated
3. Login with:
   - Username: `"admin"` (fixed)
   - Password: Master key (generated on first proxy run)
4. Proxy validates credentials and issues JWT tokens
5. Admin JWT token stored in browser localStorage
6. All subsequent UI operations use admin JWT token

**Endpoints:**
- `POST /api/auth/admin/login` - Admin login
- Admin JWT token used for all management operations

### End-User Authentication

**Flow:**
1. End-user receives credentials from admin:
   - `userId` (e.g., `"user-abc123"`)
   - `password` (generated by admin)
2. SDK calls `POST /api/auth/login` with credentials
3. Proxy validates and issues JWT tokens (access + refresh)
4. SDK stores tokens and uses for subsequent requests
5. Tokens auto-refresh when expired

**Endpoints:**
- `POST /api/auth/login` - User login (userId + password)
- `POST /api/auth/refresh` - Refresh access token
- `POST /api/auth/logout` - Revoke refresh token

### Chat Completions Authorization

**Required Headers:**
- `Authorization: Bearer {accessToken}` - JWT token (authenticates user)
- `X-API-Key: {apiKey}` - API key (identifies which key is used)

**Flow:**
1. SDK sends request with both JWT token and API key
2. Proxy validates JWT token (extracts userId)
3. Proxy validates API key format and extracts userId
4. Proxy ensures userIds match
5. Proxy checks API key status (active/revoked) in database
6. Proxy makes authorization request to PDP:
   - Principal: `User::"{userId}"`
   - Action: `Action::"completion"`
   - Resource: `Resource::"{model}"`
   - Context: Request context (IP, time, model, etc.)
7. If authorized, proxy retrieves provider API key and makes LLM request
8. Proxy returns normalized LLM response

## Data Storage

### Local SQLite Database

**Location**: `~/.zs-ai/proxy.db` (or `%APPDATA%\zs-ai\proxy.db` on Windows)

**Tables:**

1. **users**
   - `user_id` (TEXT, UNIQUE) - Generated userId (e.g., "user-abc123")
   - `email` (TEXT, UNIQUE)
   - `name` (TEXT)
   - `password_hash` (TEXT) - bcrypt hashed password
   - `role` (TEXT, optional)
   - `department` (TEXT, optional)
   - `status` (TEXT) - 'active', 'inactive', 'suspended'
   - `created_at`, `updated_at`, `last_login`

2. **api_keys**
   - `user_id` (TEXT) - References users.user_id
   - `api_key` (TEXT, UNIQUE) - Format: `sk-zs-{userId}-{hash}`
   - `key_hash` (TEXT) - SHA-256 hash for validation
   - `status` (TEXT) - 'active', 'revoked'
   - `created_at`, `revoked_at`

3. **provider_keys**
   - `provider_name` (TEXT, UNIQUE) - e.g., "openai", "anthropic"
   - `encrypted_api_key` (TEXT) - AES-256-GCM encrypted with master key
   - `metadata` (TEXT) - JSON: baseUrl, models, etc.
   - `created_at`, `updated_at`

4. **refresh_tokens**
   - `user_id` (TEXT) - References users.user_id
   - `token_hash` (TEXT, UNIQUE) - SHA-256 hash of refresh token
   - `expires_at` (DATETIME)
   - `created_at`, `revoked_at`

### Backend API (External)

**URL**: `https://api-dev.authzebra.com` (fixed, not configurable)

**Stores:**
- **Entities**: Cedar entities representing users (created when API keys are created)
- **Policies**: Cedar policies (rules) for authorization
- **Schema**: Cedar schema definition

**Note**: Users in Proxy SQLite and Entities in Backend API are **separate systems**. They're linked by `userId` but don't sync. Users are for local management, entities are for authorization.

## Master Key System

### Generation
- Generated on **first proxy run**
- Cryptographically secure random string (32+ bytes)
- Saved to: `~/.zs-ai/master-key.txt` (or `%APPDATA%\zs-ai\master-key.txt`)
- Can also be set via environment variable: `ZS_AI_MASTER_KEY`

### Usage
- **Required for ALL admin operations**
- Admin login uses master key as password
- Used to encrypt/decrypt provider API keys
- Protects database access (only master key holders can perform admin operations)

### Display
- Shown in proxy startup logs on first run
- Warning: "Save this master key - it won't be shown again!"
- Can be retrieved from config file if needed

## API Endpoints

### Proxy Server Endpoints

#### Authentication
- `POST /api/auth/admin/login` - Admin login (username: "admin", password: master key)
- `POST /api/auth/login` - End-user login (userId + password)
- `POST /api/auth/refresh` - Refresh access token
- `POST /api/auth/logout` - Revoke refresh token

#### User Management (Admin Only)
- `GET /api/users` - List all users
- `POST /api/users` - Create user (generates userId + password)
- `GET /api/users/:userId` - Get user details
- `PUT /api/users/:userId` - Update user
- `DELETE /api/users/:userId` - Delete user
- `POST /api/users/:userId/reset-password` - Reset user password

#### Key Management (Admin Only)
- `GET /api/keys` - List all API keys
- `GET /api/keys/user/:userId` - Get keys for specific user
- `POST /api/keys` - Create API key (creates entity in Backend API)
- `POST /api/keys/:userId/rotate` - Rotate key (generates new key, revokes old ones)
- `DELETE /api/keys/:userId` - Delete all keys for user (deletes from proxy DB and Backend API)

#### Provider Management (Admin Only)
- `GET /api/providers` - List all providers
- `POST /api/providers` - Add/update provider (encrypts API key)
- `DELETE /api/providers/:name` - Delete provider
- `POST /api/providers/:name/test` - Test provider connection

#### Chat Completions (End-User)
- `POST /api/chat/completions` - Make LLM request
  - Requires: `Authorization: Bearer {jwt-token}` and `X-API-Key: {api-key}`
  - Body: `{ provider, model, messages, ... }`
  - Returns: Normalized LLM response

### Backend API Endpoints (External)

**Base URL**: `https://api-dev.authzebra.com/api/v2025-01/projects/{projectId}`

#### Authentication
- `POST /oauth2/token` - Get M2M token (client_credentials grant)

#### Entities
- `GET /entities` - List entities (with optional `uid` query param)
- `POST /entity` - Create entity (body: `{ entity: {...}, status: 1 }`)
- `PUT /entity` - Update entity (body: `{ entity: {...}, status: 2 }` for revoked)
- `DELETE /entity?entityName=User::"userId"` - Delete entity

#### Policies
- `GET /policies` - List policies
- `POST /policies` - Create policy
- `DELETE /policies` - Delete policy

#### Schema
- `GET /schema` - Get Cedar schema
- `POST /schema` - Update Cedar schema

**Note**: All Backend API requests require:
- `Authorization: Bearer {m2m-token}`
- `x-project-id: {projectId}`
- `x-op-id: {operation-id}`
- `x-session-id: {session-id}`

## Key Management Operations

### Create Key
1. Admin selects user from dropdown in UI
2. Sets spend limits (optional)
3. Proxy generates API key: `sk-zs-{userId}-{hash}`
4. Proxy stores key in SQLite database
5. Proxy creates entity in Backend API with:
   - UID: `User::"{userId}"`
   - Attributes: user details, spend limits, etc.
   - Status: `1` (active)
6. API key returned to admin (shown once)

### Rotate Key
1. Admin clicks "Rotate" for a user
2. Proxy revokes all existing active keys for that user (sets status to 'revoked')
3. Proxy generates new API key for same user
4. Proxy stores new key in database
5. **Note**: Entity in Backend API is NOT updated (same userId, new key in proxy only)
6. New API key returned to admin (shown once)

### Delete Key
1. Admin clicks "Delete" and confirms
2. Proxy deletes entity from Backend API: `DELETE /entity?entityName=User::"{userId}"`
3. Proxy deletes all keys for user from SQLite database
4. Success confirmation shown

**Note**: There is no "revoke" operation anymore - only rotate and delete.

## Port Management

### Auto-Selection
- Default port: `3100`
- If port in use, tries next port: `3101`, `3102`, etc.
- Continues until finding free port
- Selected port displayed in startup logs
- Port stored in config file

### Server Binding
- Listens on `127.0.0.1` (IPv4 localhost)
- UI and API served on same port
- UI accessible at: `http://localhost:{port}`
- API accessible at: `http://localhost:{port}/api/*`

## PDP Authorization

### Request Format

**Endpoint**: `{pdpUrl}/authorize`

**Request Body:**
```json
{
  "principal": "User::\"user-abc123\"",
  "action": "Action::\"completion\"",
  "resource": "Resource::\"gpt-4\"",
  "context": {
    "ip_address": {
      "__extn": {
        "fn": "ip",
        "arg": "192.168.1.1"
      }
    },
    "request_time": "2025-01-13T10:30:00Z",
    "hour": 10,
    "model_name": "gpt-4",
    "is_emergency": false,
    "day_of_week": "Monday"
  }
}
```

**Response:**
```json
{
  "decision": "Allow" | "Deny",
  "diagnostics": {
    "reason": ["..."],
    "errors": []
  }
}
```

**Note**: PDP already has all entities, policies, and schema. Proxy just makes authorization requests.

## Key Implementation Details

### API Key Format
- Format: `sk-zs-{userId}-{hash}`
- Example: `sk-zs-user-abc123-4b9d643d0e5c888b`
- User ID extraction: Finds last hyphen before hash
- Supports hyphenated user IDs

### Entity Creation
- When API key is created, entity is created in Backend API
- Entity UID: `User::"{userId}"`
- Entity attributes include user details, spend limits, etc.
- Entity status: `1` (active) or `2` (revoked)
- Entity update: Full entity body required (same as create)

### Entity Deletion
- DELETE request: `DELETE /entity?entityName=User::"{userId}"`
- Deletes entity from Backend API
- Also deletes keys from proxy database

### Cedar Extension Types
- **Decimal**: `{ "__extn": { "fn": "decimal", "arg": "10.50" } }`
- **IP Address**: `{ "__extn": { "fn": "ip", "arg": "192.168.1.1" } }`

### PDP Request Format
- Principal: `User::"{userId}"` (Cedar entity UID format)
- Action: `Action::"completion"` (Cedar entity UID format)
- Resource: `Resource::"{model}"` (Cedar entity UID format)
- Context: Request context with Cedar extension types

### Provider API Key Encryption
- Uses AES-256-GCM encryption
- Master key used for encryption/decryption
- Encrypted keys stored in SQLite database
- Decrypted on-the-fly when making LLM requests

### JWT Token Details
- Access token: 1 hour expiry
- Refresh token: 7 days expiry
- Signed with master key
- Claims: `userId`, `email`, `role`, `name`

## Important Notes

1. **No Revoke Operation**: Only rotate and delete are available. Rotate generates a new key and revokes old ones. Delete removes the entity entirely.

2. **Rules #active Removed**: All policies are treated as active. No #active tag checking in UI or code.

3. **Admin Login**: Fixed username "admin", password is master key. No separate admin user in database.

4. **Backend URL Fixed**: Always `https://api-dev.authzebra.com`, not configurable.

5. **Auth Mode Fixed**: Always "live" mode (PDP only), not configurable.

6. **Users vs Entities**: Users in Proxy SQLite are separate from Entities in Backend API. They're linked by userId but don't sync.

7. **Master Key**: Required for all admin operations. Generated on first run, saved to file.

8. **Port Auto-Selection**: Starts from 3100, increments if port in use.

9. **UI Served by Proxy**: UI and API on same port. UI is static files served by Express.

10. **Nuxt Server API Routes**: In dev mode, Nuxt server API routes proxy requests to proxy server. In production, UI is static files.

## Security Considerations

1. **Master Key**: Critical - protects all admin operations and encrypted data
2. **JWT Tokens**: Short-lived access tokens, long-lived refresh tokens
3. **API Keys**: Validated on every request, can be revoked
4. **Provider Keys**: Encrypted at rest with master key
5. **Password Storage**: bcrypt hashed, never stored in plain text
6. **Database**: SQLite file protected by master key requirement
